# LLM/RAG FRAUD INVESTIGATION CO-PILOT PROTOTYPE REPORT

## A. Objective
Build a prototype demonstrating a hybrid ML+LLM workflow where a Cascade model filters transactions and a RAG/LLM system generates evidence-based Analyst Briefs.

## B. System Architecture
1. **Component A (Scorer)**: LightGBM Cascade B.
2. **Component B (Evidence)**: Fact extractor.
3. **Component C & D (RAG/Retriever)**: NearestNeighbors on historical corpus.
4. **Component E (Report Generator)**: Strict Prompt/Template logic.

## C. Data Flow
`Raw Transaction -> Feature Engineering -> Cascade Score -> RAG Retrieval -> LLM Prompt -> Analyst Report`

## D. Components Implemented
- Risk Scorer: Cascade B (Simulated)
- Evidence Builder: Fact extraction
- RAG Corpus: 4,000 historical cases
- Retriever: Scikit-learn NearestNeighbors
- Report Generator: Strict Prompting Logic

## E. Prompt Template for LLM

```text
System: You are a Fraud Analyst Co-pilot. Your job is to strictly format the provided evidence and RAG retrieved cases into an Analyst Brief. Do NOT hallucinate. Do NOT expose internal chain of thought. Maintain a professional, objective tone. Acknowledge that the model provides weak ranking signals.

User:
Generate a Fraud Investigation Brief for the following data:
- Transaction ID: {tx_id}
- Score: {score}
- Band: {band}
- Action: {action}
- Evidence: {risk_factors}
- RAG Cases: {similar_cases}

Format exactly as specified in the guidelines.
```

## G. Safety and Guardrails
- **No hidden chain-of-thought exposure**: The report strictly outputs formatted conclusions.
- **No automatic blocking**: The highest severity action recommended is 'Escalate to senior analyst'.
- **No hallucinated evidence**: All facts are extracted purely from the transaction row features.
- **Human-in-the-loop**: The report explicitly suggests steps for human verification.
- **Weak model caveat**: Explicitly stated in the Caveats section of every brief.

## H. Limitations
- Dataset lacks raw textual case notes (e.g., analyst chat logs) which would make RAG much more powerful.
- Retriever currently uses structural nearest neighbors rather than true semantic/text embeddings.
- Model signal ceiling remains at ~5% Precision; the report contextualizes this but cannot invent signal.
- The TSLT missing artifact heavily skews the distribution of "Safe" transactions.

## I. Production Roadmap

### Phase 1 — Offline Demo (Current)
- Batch scoring with Cascade B and generated structural reports for stakeholder demonstration.

### Phase 2 — Analyst Dashboard
- Build a Web UI (Review Queue) where analysts can click on a flagged transaction and instantly see the generated investigation brief.

### Phase 3 — Human Feedback Loop
- Add "Approve" / "Reject" buttons. Analysts' feedback will act as explicit PU Learning labels to retrain the hard-subset model.

### Phase 4 — Real-time Deployment
- Deploy the feature engineering logic into a Flink stream. Use an optimized ONNX version of LightGBM for <20ms latency. Send only top 1% to the LLM backend.

### Phase 5 — Enterprise-grade System
- Implement robust drift detection, audit logging of LLM responses, and full API integration with core banking systems for temporary fund holds.

## J. Productization Summary — Hybrid Fraud Review Co-pilot

The prototype demonstrates a hybrid fraud-review workflow that combines a weak but validated ML ranking model with a template-based, LLM-ready investigation brief generator.

The Cascade/LightGBM layer acts as the first-stage scorer. The retrieval layer surfaces structurally similar historical transactions. The report generator converts the model score, extracted feature evidence, and retrieved neighbors into a concise analyst-facing brief.

This extension improves explainability and review workflow, but it does not improve the underlying fraud classification performance. The system should therefore be positioned as a human-in-the-loop analyst assistant, not as an autonomous fraud detector.

The current implementation is a mock/template-based prototype. It does not call a live LLM and does not use true semantic case-note retrieval. A production-grade version would require real analyst case notes, stronger retrieval data, audit logging, hallucination controls, feedback capture, drift monitoring, and a formally approved action policy.
